# Saved Session Rendering Fix - Complete

## Root Cause Identified

The Hermes overlay's saved-session rendering was broken because `_history_to_messages()` in `tui_gateway/server.py` was NOT properly transforming stored tool results into human-readable format.

### The Data Flow

**Live Streaming:**
```
Agent → WebSocket events (raw text deltas) → TUI client renders via markdown.tsx
```

**Saved Session (FIXED):**
```
SQLite DB → get_messages_as_conversation() → _history_to_messages() → _coerce_message_text() → _summarize_tool_result() → TUI client renders
```

## Bugs Fixed

### 1. Sentinel-Prefixed JSON Not Handled (CRITICAL)

**Problem:** Tool results stored as `\x00json:{"total_count": 0}` were returned as-is instead of being parsed and summarized.

**Fix:** Added handler in `_coerce_message_text()`:
```python
if content.startswith("\x00json:"):
    parsed = json.loads(content[6:])  # Strip 6-char prefix
    return _summarize_tool_result(parsed)
```

### 2. Falsy Zero Values Lost in `or` Chains (CRITICAL)

**Problem:** `total = data.get('total_count') or data.get('count')` treated `total_count=0` as falsy, resulting in `None`.

**Fix:** Use explicit `is not None` checks:
```python
total = data.get('total_count') if data.get('total_count') is not None else (
    data.get('count') if data.get('count') is not None else data.get('total')
)
```

### 3. Tool Result Text Not Included in Output (CRITICAL)

**Problem:** Tool messages created as `{role, name, context}` without the summarized `text` field.

**Fix:** Include `content_text` in tool message:
```python
messages.append({
    "role": "tool", 
    "name": name, 
    "context": _tool_ctx(name, args), 
    "text": content_text  # ← ADDED
})
```

### 4. Indentation Errors (Introduced During Patching)

**Problem:** Multiple patch attempts corrupted function indentation.

**Fix:** Rewrote entire `_summarize_tool_result()` function with correct 4-space indentation.

## Why Diffs Are Not Stored

Code diffs (unified diff format with +/- highlighting) are **live-only enhancements**:
- Generated by `render_edit_diff_with_delta()` during tool execution
- Sent via `tool.complete` event's `inline_diff` field
- NOT persisted to database
- On resume, you see the tool result ("File updated") but not the animated diff

This is **correct behavior** - diffs are a visualization of the change process, not permanent transcript content. The actual file changes ARE persisted (via the tool's effect on the filesystem).
